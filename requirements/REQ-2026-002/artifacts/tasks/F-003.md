# F-003 · 关闭 H1/H3/H4/H5 四个高优先级缺口

## 需求描述

- **H1 R005 事务化**：review_verdict plugin 改造为 staged_writes 模式（drift 检测仅 append `ctx.staged_writes`，runner 在所有 gate pass 后调 commit_staged_writes 落盘；fail 路径直接丢弃）。覆盖三路径单测。
- **H3 submit 与 next 同源**：`/requirement:submit` 改写为调用 `run.py --trigger=submit`，复用 phase-transition gate 集合。
- **H4 PR 合并状态闭环**：实现 plugins/pr_state.py（gh pr view 查 MERGED 状态），加入 submit trigger。
- **H5 Bash 写保护**：实现 plugins/bash_write_protect.py（正则覆盖 8+ shell 写法 + 双轨白名单：父进程链识别 save-review.sh + `CLAUDE_GATES_BYPASS` 环境变量 + reason 强制），改写 protect-branch.sh 切到 runner 的 pre-tool-use trigger。

## 接口/数据结构

详见 `requirements/REQ-2026-002/artifacts/detailed-design.md`：
- §3.1 review_verdict staged_writes 模式（含 commit_staged_writes 时序）
- §3.2 pr_state plugin 接口
- §3.3 bash_write_protect 正则（heredoc / printf / dd / mv / cp / cat>/tee/python -c）
- §4.2 submit 时序（复用 phase-transition gate）
- §4.3 pre-tool-use 时序

接口已在 detail-design 阶段冻结（`interfaces_frozen: true`）。

## 验收

详见 `features.json` F-003.acceptance：
1. H1 单测三路径全过：drift→pass→commit / drift→fail→rollback / r005-only-fail→no-write
2. H3：submit 与 next 在同输入下 `Report.passed` / `Report.failed` 集合相同
3. H4：PR 已 MERGED 时 submit 阻断并提示 `/requirement:next`；未 MERGED 时正常推进
4. H5：Bash 写 reviews/*.json 全形态被拦——cat >/>>、tee/tee -a、mv、cp、python3 -c open(w)、heredoc（cat <<EOF >）、printf >、dd of= 至少 8 类
5. H5：save-review.sh 子进程经父进程链识别白名单放行；`CLAUDE_GATES_BYPASS=1` 缺 `CLAUDE_GATES_BYPASS_REASON` 时拦截并给清晰提示
